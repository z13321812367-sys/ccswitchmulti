# CCSwitchMulti v3.16.4-7

v3.16.4-7 是 `v3.16.4-4` 之后的正式汇总发布，包含 `v3.16.4-5`、`v3.16.4-6` 和 `v3.16.4-7` 三段实际代码变更。这个版本重点解决 Codex MultiRouter 错路由、Codex Desktop 登录态被旧备份覆盖、OAuth Responses 请求形态、火山 AgentPlan 模型刷新，以及含图请求在部分上游被安全审核拒绝后的兜底体验。

## 重点更新

### Codex MultiRouter 路由和模型目录一致性

- 修复官方 GPT route 与第三方中转 GPT route 同时存在时，同名模型如 `gpt-5.5` 可能被 route 顺序抢走的问题。保存和同步 MultiRouter 时会为第三方同名模型生成稳定可见别名，并通过 route 级 `modelMap` 映射回真实上游模型。
- 修复 provider 模型目录临时为空时，MultiRouter 同步逻辑把已保存的第三方 relay alias、`match.models` 和 `modelMap` 清空的问题。现在目标 provider 暂无可用 catalog 时会保留已保存路由，不把短暂刷新失败当成用户删除模型。
- 修复聚合 `modelCatalog` 暴露 route 实际接不住的模型的问题。Codex 选择器现在只看到当前 route 能 exact 或 prefix 匹配的可见模型，避免用户选到一个随后落入官方前缀 route 的第三方模型。
- Provider 保存或模型刷新后，会同步更新引用它的 MultiRouter plan，包括 route 匹配模型、route 级 `modelMap`、聚合模型目录和子 Agent 候选；被删除的子 Agent 候选会提示用户手动处理。

### MultiRouter 向导

- 新增 MultiRouter 命名步骤，创建或编辑方案时可直接命名，便于在 provider 列表、状态页和后续启用提示中区分方案。
- 新增最终模型池整理步骤，可剔除旧模型或不想暴露给 Codex 的模型，并调整模型顺序。
- 新增子 Agent 候选选择步骤，可从最终模型池中选择并排序最多 5 个候选模型。
- 模型读取页改为按 provider 展示独立状态，区分等待、读取中、有更新、无变化、跳过和失败，并显示模型新增、移除或能力变化摘要。
- Codex provider 协议探测改为小并发执行，减少多模型 Responses / Chat 双协议探测等待时间，同时避免一次性打爆上游限流。

### Codex Desktop 登录态保护

- 修复接管恢复、崩溃恢复、关机恢复或接管关闭时，旧备份中的空 auth、API key auth 或过期 OAuth 快照覆盖当前 Codex Desktop `auth.json` 的问题。当前机器已有真实 ChatGPT/Codex OAuth 登录材料时，只恢复 `config.toml`，不回滚或删除 live 登录态。
- 官方 Codex provider 切换时新增账号边界判断：同一账号保留 live OAuth token，明确切换到另一个官方账号时允许写入目标账号 auth，避免“保护登录态”反过来阻止多账号切换。
- 第三方 Codex provider 的 bearer/API key 继续写入 `config.toml` 的 provider 字段，不再把第三方密钥当成官方登录材料写进 `auth.json`。

### Responses 请求形态修复

- 修复 official managed Codex OAuth `/responses` 直透 ChatGPT Codex backend 时，工具输出 item 携带冗余 `content` 字段导致 HTTP 400 的问题。`message` / `reasoning` 会保留 `content`，`function_call_output`、`custom_tool_call_output`、`tool_search_output` 等非 message item 会删除不兼容的 `content`，同时保留 `output`、`call_id`、`tools`、`status` 等有效字段。
- 修复 Codex 在同一会话内跨 provider 或跨模型切换后，`input` 中残留的 `system` / `developer` 控制消息被直接透传给第三方 Responses API 的问题。现在这些控制消息会提升到顶层 `instructions`，不再作为普通对话 input 触发上游 400。
- 以上修复限定在 Codex Responses 相关路径；Responses 转 Chat 的转换路径继续使用原有角色合并逻辑。

### 火山 AgentPlan 模型刷新

- 火山 AgentPlan 的模型枚举新增管控面 OpenAPI 路径：配置 AccessKey ID / SecretAccessKey 后，模型刷新会调用 `ListArkAgentPlanModel`，不再错误地只猜数据面 OpenAI `/models`。
- 如果火山 AgentPlan 没有 AK/SK 但有推理 API Key，仍会回落尝试数据面 `/models`，兼容可返回 OpenAI 格式模型列表的账号或网关。
- 如果火山 AgentPlan 缺少 AK/SK 和推理 API Key，会保留内置 modelCatalog 并给出 catalog-only 提示，不把“无法在线枚举”误判为模型被清空。
- BytePlus Plan 仍按 catalog-only 处理，避免在没有明确模型枚举接口契约时发起错误请求。

### 含图请求兜底

- 图片错误兜底从“仅不支持图片”扩展到可重试的图片审核/安全策略拒绝。部分上游如 MiniMax 返回 `base_resp.status_msg` 中的 sensitive / safety / policy 类错误时，CCSwitchMulti 会移除图片块后重试一次。
- 兜底仍要求请求体里确实包含图片，不会把纯文本请求的安全错误误判为图片兜底场景。

### 发布和排障

- Windows x64 发布包改为 NSIS setup `.exe`，减少 MSI / WiX 构建链路的不稳定；ARM64 仍保留对应平台打包方式。
- GitHub Release workflow 修复了 Linux history repair sidecar 预构建、Windows LLVM 缺失回退安装、macOS 无签名密钥时降级为 unsigned bundle 等问题。
- 新增 Codex OAuth/native request 诊断脚本和请求形态对比工具，用于定位官方容量、OAuth token、本地代理转发和 request shape 差异。
- 新增 Windows history repair tool 打包脚本，方便把 standalone 历史修复工具制作为单文件可执行程序。

## 相比 v3.16.4-4 的版本分段

- `v3.16.4-5`：合入登录态保护、MultiRouter 向导重构、provider 编辑后 MultiRouter 自动同步、并发协议探测、发布 workflow 修复和排障工具。
- `v3.16.4-6`：修复 official Codex OAuth `/responses` 中非 message item 多余 `content` 字段导致的 HTTP 400。
- `v3.16.4-7`：修复 MultiRouter 第三方 GPT alias 错路由、不可路由 catalog 暴露、火山 AgentPlan 模型枚举、敏感图片兜底重试，以及第三方 Responses 控制消息透传问题。

## 已知边界

- 本版本不会凭空创建上游不存在的模型。若第三方平台 `/models` 没有返回某个模型，例如 `gpt-5.5-pro`，CCSwitchMulti 不会合成该模型名；旧机器上手工添加或历史残留的模型仍需要在 MultiRouter 模型整理步骤中确认。
- 重复 exact route 在运行时仍然按 route 顺序选择第一条；本版本的修复重点是在保存和同步阶段生成唯一可见别名，避免正常配置进入重复 exact 状态。
- macOS / Linux 资产仍依赖对应平台构建或 GitHub Actions 补充；Windows 本地发布路径产出 NSIS installer、portable zip、raw exe、`latest.json` 和 SHA256。
